# url_shortcut

Сервис заменяет внешние ссылки короткими ссылками нашего домена, чтобы сайт мог
безопасно отдавать переходы через собственный REST API и собирать статистику.

## План решения

### Goal
Собрать REST-сервис на Spring Boot 2, который:
- регистрирует сайты;
- выдает логин и пароль;
- авторизует через JWT;
- преобразует исходный URL в короткий код;
- делает публичный redirect по коду;
- возвращает статистику по переходам.

### Done
Задача считается завершенной, когда:
- `POST /registration` регистрирует новый сайт и не дублирует уже существующий;
- `POST /login` возвращает JWT по `login/password`;
- `POST /convert` доступен только с `Bearer` токеном и возвращает уникальный код;
- `GET /redirect/{code}` без авторизации отвечает `302` и выставляет `Location`;
- `GET /statistic` возвращает список `{url, total}`;
- счетчик переходов увеличивается атомарно в базе данных.

### Constraints
- использовать Spring Boot 2;
- сервис работает через REST API;
- авторизация только через JWT;
- увеличение счетчика делать в БД, а не в Java;
- двигаться маленькими end-to-end срезами с проверкой после каждого.

### Non-goals
- UI или админ-панель;
- срок жизни коротких ссылок;
- кастомные домены, rate limit, QR-коды;
- production-инфраструктура и деплой.

### Facts
- в условии 5 пользовательских сценариев: регистрация сайта, авторизация,
  конвертация URL, redirect и статистика;
- redirect должен работать без авторизации;
- сервис выдает парные `login/password` для сайта, а не для конечного
  пользователя;
- в статистике нужен `url` и общее количество переходов;
- сейчас в репозитории нет реализации, только базовый `README`.

### Assumptions
- сайт идентифицируется по нормализованному URL или host и не должен
  регистрироваться повторно;
- короткий код генерируется на сервере и уникален в пределах всей системы;
- статистика нужна по ссылкам, зарегистрированным текущим сайтом;
- для хранения подойдет реляционная БД, где можно сделать атомарный
  `update ... set total = total + 1`.

### Open questions
- что считать уникальностью сайта: полный URL или только host;
- нужно ли переиспользовать код для одинакового URL одного сайта;
- какой статус отдавать при неизвестном коде: `404` или другой;
- требуется ли сортировка статистики по `total` или достаточно произвольного
  порядка.

## Thin slices

### Slice 1. Регистрация сайта
Первый пользовательский результат: сервис принимает URL сайта, сохраняет его и
возвращает `registration`, `login`, `password`.

Что реализовать:
- каркас Spring Boot 2 приложения;
- таблицу/сущность сайта;
- генерацию уникальных `login/password`;
- проверку повторной регистрации.

Checkpoint command:
```bash
./mvnw test -Dtest=SiteRegistrationControllerTest
```

Expected result:
- новый сайт регистрируется успешно;
- повторная регистрация того же сайта возвращает `registration=false`;
- данные сайта реально сохраняются в БД.

### Slice 2. Авторизация и защищенный convert
Второй результат: зарегистрированный сайт получает JWT и может вызвать
`POST /convert`.

Что реализовать:
- `POST /login` по `login/password`;
- Spring Security + JWT filter;
- таблицу/сущность сокращенных ссылок;
- `POST /convert` с выдачей уникального кода.

Checkpoint command:
```bash
./mvnw test -Dtest=AuthControllerTest,ConvertControllerTest
```

Expected result:
- валидные credentials дают JWT;
- без токена `POST /convert` получает `401`;
- с токеном сервис возвращает код и связывает его с исходным URL и сайтом.

### Slice 3. Redirect и атомарный счетчик
Третий результат: короткая ссылка ведет на исходный URL, а количество переходов
увеличивается без гонок.

Что реализовать:
- `GET /redirect/{code}` без авторизации;
- поиск исходного URL по коду;
- ответ `302` + `Location`;
- SQL-обновление счетчика в БД.

Почему счетчик должен расти в БД:
- при конкурентных переходах Java-логика вида `read -> increment -> save`
  приводит к lost update;
- атомарный `update total = total + 1` выполняется ближе к данным и не теряет
  инкременты.

Checkpoint command:
```bash
./mvnw test -Dtest=RedirectControllerTest,ShortcutStatisticRepositoryTest
```

Expected result:
- существующий код отдает `302`;
- отсутствующий код дает предсказуемую ошибку, например `404`;
- повторные переходы корректно увеличивают `total`.

### Slice 4. Статистика и финальная сборка
Финальный результат: авторизованный сайт видит статистику своих ссылок и проект
готов к сдаче.

Что реализовать:
- `GET /statistic`;
- DTO для ответа `{url, total}`;
- сортировку статистики, если она нужна;
- минимальную документацию по запуску и smoke-check через `curl`.

Checkpoint command:
```bash
./mvnw test
```

Expected result:
- весь набор тестов зеленый;
- статистика возвращает ссылки и числа переходов;
- сценарий из задания проходится вручную от регистрации до redirect.

## Минимальная модель данных

`site`
- `id`
- `site`
- `login`
- `password`

`shortcut`
- `id`
- `code`
- `url`
- `site_id`
- `total`

## Порядок реализации
1. Сначала закрыть регистрацию сайта и persistence-слой.
2. Затем добавить JWT и защитить `convert`/`statistic`.
3. После этого сделать redirect с атомарным обновлением `total`.
4. В конце собрать статистику, smoke-тесты и описание API.
